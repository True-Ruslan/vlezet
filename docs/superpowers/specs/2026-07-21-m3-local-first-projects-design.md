# M3 — Local-First Projects Design

## Status

Approved for implementation on 2026-07-21.

## Product outcome

M3 turns Vlezet from a session-only editor into a dependable personal product. A user can maintain several apartment-layout projects, close and reopen the browser without losing work, create alternative variants, transfer a project through a file, and export a clean plan image.

The milestone remains local-first. It does not introduce accounts, a backend, cloud sync, collaboration, public links, billing, PDF generation, or version history.

## User promise

> Your apartment plan is saved automatically on this device, can be backed up as a Vlezet file, and can be exported as a clean image.

The product must never imply cloud backup. UI copy must explicitly say that projects are stored in this browser on this device.

## Chosen approach

Vlezet uses IndexedDB through a framework-independent project repository interface.

The application has four separated layers:

1. **Project model and file format** — pure TypeScript types, validation, migrations, naming and cloning.
2. **Project repository port** — asynchronous CRUD interface independent from browser APIs.
3. **IndexedDB adapter** — browser implementation with atomic transactions.
4. **Project session UI** — dashboard, active-project lifecycle, autosave status, import/export and editor integration.

The existing `VlezetDocument` remains the only source of truth for apartment geometry. Project metadata wraps it but never duplicates walls, rooms, openings or placed objects.

## Scope

### Included

- project dashboard;
- create, open, rename, duplicate and delete project;
- remember and restore the last opened project;
- automatic document persistence after semantic editor operations;
- throttled viewport persistence;
- save status: loading, saved, saving, failed;
- IndexedDB storage with an in-memory repository for tests;
- `.vlezet.json` import and export;
- migration of schema v1/v2/v3 documents during import;
- clean PNG export from the current plan;
- fit-to-content command;
- collapsible furniture catalogue;
- project name and save state in the toolbar;
- non-blocking toast and confirmation UI;
- recovery UI for unreadable project data;
- explicit local-storage disclosure;
- M3 browser acceptance document.

### Excluded

- authentication;
- PostgreSQL or server API;
- multi-device synchronization;
- collaborative editing;
- public sharing links;
- project revision history;
- PDF export;
- plan image/PDF tracing;
- 3D;
- AI suggestions.

## Project record

```ts
export type ProjectViewport = Readonly<{
  offsetX: number;
  offsetY: number;
  pixelsPerMillimeter: number;
}>;

export type ProjectUiState = Readonly<{
  furnitureCatalogOpen: boolean;
}>;

export type VlezetProjectRecord = Readonly<{
  storageVersion: 1;
  id: string;
  name: string;
  createdAt: string;
  updatedAt: string;
  document: VlezetDocument;
  viewport: ProjectViewport;
  ui: ProjectUiState;
}>;
```

Rules:

- timestamps are ISO-8601 UTC strings;
- IDs are UUIDs generated by the application;
- project name is trimmed, 1–80 characters;
- viewport scale is finite and clamped to the same editor limits, `0.01..2` pixels/mm;
- offsets are finite numbers;
- `document` is validated and migrated to the current domain schema before entering the editor;
- project records are immutable values at module boundaries.

## Storage database

Database name: `vlezet`.

Database version: `1`.

Object stores:

### `projects`

- key path: `id`;
- index: `updatedAt`;
- stores complete `VlezetProjectRecord` values.

### `settings`

Key/value records:

```ts
{ key: "lastProjectId", value: string | null }
```

No editor history is persisted. Opening a project creates a fresh history state around the stored current document.

## Repository interface

```ts
export interface ProjectRepository {
  list(): Promise<readonly VlezetProjectRecord[]>;
  get(id: string): Promise<VlezetProjectRecord | null>;
  put(project: VlezetProjectRecord): Promise<void>;
  delete(id: string): Promise<void>;
  getLastProjectId(): Promise<string | null>;
  setLastProjectId(id: string | null): Promise<void>;
}
```

`list()` returns newest projects first by `updatedAt`, with `id` as a deterministic secondary key.

The IndexedDB adapter must use one read-write transaction per mutation. `put(project)` validates the complete record before writing it.

An in-memory adapter with identical semantics is required for unit tests.

## Project operations

Pure functions provide deterministic behavior:

```ts
createProject(input)
renameProject(project, name, now)
duplicateProject(project, newId, now)
replaceProjectDocument(project, document, now)
replaceProjectViewport(project, viewport, now)
replaceProjectUi(project, ui, now)
```

Duplication rules:

- duplicate receives a new project ID;
- document entity IDs remain unchanged because they are scoped to a project;
- name becomes `«Original name — копия»`, trimmed to 80 characters;
- `createdAt` and `updatedAt` become the duplication time;
- viewport and UI state are copied.

## Import file format

Extension: `.vlezet.json`.

MIME type: `application/json`.

```ts
export type VlezetProjectFileV1 = Readonly<{
  format: "vlezet-project";
  fileVersion: 1;
  exportedAt: string;
  project: Readonly<{
    name: string;
    document: VlezetDocumentV1 | VlezetDocumentV2 | VlezetDocumentV3;
    viewport?: ProjectViewport;
  }>;
}>;
```

Import behavior:

1. Parse UTF-8 JSON.
2. Require `format === "vlezet-project"` and `fileVersion === 1`.
3. Validate project name, viewport and document shape.
4. Migrate document to schema v3.
5. Generate a new local project ID.
6. Set fresh `createdAt` and `updatedAt`.
7. Use default viewport if omitted.
8. Save before opening.

A malformed file must not modify existing projects or the current editor session. Error messages distinguish invalid JSON, unsupported file version and invalid project data.

Export behavior:

- exports the latest saved in-memory project state, not a stale repository read;
- filename is a filesystem-safe slug plus `.vlezet.json`;
- exported JSON is pretty-printed with two spaces;
- no undo history, selection or transient preview is exported.

## Autosave semantics

### Document

The editor store exposes a subscription to the current `history.document` reference.

A completed semantic command changes that reference exactly once. The session schedules persistence with a 150 ms debounce. A later semantic operation replaces the pending snapshot. The repository write always receives a fully validated project.

### Viewport

Viewport changes are persisted with a 500 ms throttle and a trailing write. Pan and zoom do not create editor history entries.

### UI preferences

Furniture catalogue open/closed state is saved with the next project write.

### Status state machine

```ts
type SaveStatus =
  | { kind: "idle" }
  | { kind: "saving" }
  | { kind: "saved"; savedAt: string }
  | { kind: "failed"; message: string };
```

Rules:

- opening a project begins in `idle` and transitions to `saved` after load;
- pending document changes immediately show `saving`;
- successful write shows `saved` with time;
- failed write preserves the dirty in-memory project and shows retry;
- retry writes the newest snapshot, not the failed old snapshot;
- a later successful write clears the failure.

### Page lifecycle

Data safety does not depend on `beforeunload` because IndexedDB writes are asynchronous and browsers may terminate them.

The app writes shortly after each semantic operation. `pagehide` performs a best-effort flush only when a dirty snapshot remains.

## Session lifecycle

Application modes:

```ts
type AppMode =
  | { kind: "loading" }
  | { kind: "dashboard" }
  | { kind: "editor"; projectId: string }
  | { kind: "recovery"; projectId: string; error: string };
```

Startup:

1. Open IndexedDB.
2. Read `lastProjectId`.
3. If it resolves to a valid project, open it.
4. Otherwise show dashboard.
5. If a stored record exists but fails validation, show recovery mode without overwriting or deleting it.

Opening another project flushes the current dirty snapshot before switching. If flushing fails, the user sees a modal with `Повторить` and `Открыть без сохранения`. The destructive option is explicit.

## Dashboard UX

Dashboard content:

- Vlezet branding and short local-storage explanation;
- primary `Новый проект` button;
- `Импортировать проект` action;
- cards sorted by most recently changed;
- project name;
- updated relative time and exact timestamp tooltip;
- counts for rooms, walls and placed objects;
- `Открыть`, `Дублировать`, `Переименовать`, `Удалить` actions;
- empty state with one clear call to action.

Deletion requires a modal naming the project. The default focused action is `Отмена`, not deletion.

## Editor integration

### Loading a project

`editorStore` gains:

```ts
loadDocument(document: VlezetDocument): void;
```

It replaces history with `createHistoryState(document)` and clears all selection, draft and gesture state.

The canvas receives initial viewport from project session and publishes viewport changes through a callback/store port.

### Toolbar

The toolbar shows:

- back-to-projects button;
- editable project name;
- save-state text/icon;
- `Показать весь план`;
- `Экспорт` menu with PNG and Vlezet JSON;
- existing drawing tools and history controls.

The project name commits on Enter or blur, validates locally, and does not create editor undo history.

### Furniture catalogue

`Мебель (F)` toggles the entire catalogue panel. The current shortcut no longer merely focuses the first card.

The panel state persists per project.

## Fit to content

A pure geometry helper calculates world bounds from:

- wall centerline endpoints expanded by half wall thickness;
- room polygons;
- opening swing geometry;
- placed-object footprints and clearance zones only when requested for export.

Interactive fit uses physical content without clearance zones.

The viewport is calculated with:

- 64 px minimum padding on every side;
- scale clamped to `0.01..2`;
- empty-document fallback to the default viewport;
- deterministic center alignment.

## PNG export

PNG export uses a dedicated export renderer, not a screenshot of UI chrome.

The renderer receives only:

```ts
renderPlanPng({ document, pixelRatio, paddingPx, includeRoomLabels })
```

Output requirements:

- white opaque background;
- automatic content bounds;
- walls with physical thickness and real gaps for openings;
- doors, windows and swing arcs;
- room fills, names and areas;
- furniture footprints and labels;
- no selection, Transformer, snapping guides, clearance overlays, grid, inspector or catalogue;
- default pixel ratio `2`;
- maximum output dimension `8192` px; scale down proportionally when exceeded;
- downloaded filename uses project slug plus `.png`.

The implementation may use an off-screen Konva stage because Konva is already the rendering dependency. Export layout calculations remain pure and testable.

## Error handling

No workflow uses `window.alert` or `window.confirm`.

Errors appear through:

- inline dashboard/import messages;
- save-status failure with retry;
- modal for destructive or project-switch decisions;
- non-blocking toast for completed export/import/duplicate actions.

Raw IndexedDB, JSON parser and browser exception messages are not shown directly. They are mapped to Russian product copy while logged to `console.error` in development.

## Accessibility

- dashboard and modal actions are keyboard reachable;
- modal traps focus and closes with Escape when safe;
- destructive action has an explicit accessible name including project name;
- save status uses `aria-live="polite"`;
- errors use `role="alert"`;
- import uses a labelled file input accepting `.json,.vlezet.json`;
- export menu is operable by keyboard.

## Acceptance criteria

M3 is complete when all of the following are true:

1. A user creates two named projects and sees both after a full browser reload.
2. Editing walls, openings, room names and furniture persists automatically.
3. Reopening a project restores the document and viewport.
4. Duplicate creates an independent alternative layout.
5. Deleting one project cannot delete or mutate another.
6. Exported `.vlezet.json` imports into a new independent project with equivalent document geometry.
7. Invalid import leaves all existing data unchanged and shows a useful error.
8. PNG contains only the plan at readable resolution and excludes editor chrome.
9. Furniture panel can be hidden and restored with the button and `F`.
10. `Показать весь план` frames all physical content.
11. A simulated repository failure keeps the current in-memory work and supports retry.
12. `pnpm test`, `pnpm typecheck`, `pnpm lint` and `pnpm build` all pass in one clean CI run.
