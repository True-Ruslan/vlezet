# M4.5 — Assisted Recognition browser acceptance

Цель проверки: убедиться, что распознавание ускоряет обводку реального плана, но никогда не подменяет собой точную геометрию Vlezet и не портит существующий проект.

## 1. Подготовка

1. Запустить ветку `feat/m4-5-assisted-recognition`.
2. Создать новый проект из JPG/PNG/PDF-плана.
3. Откалибровать масштаб по известному размеру.
4. Убедиться, что появилась кнопка `Распознать ✦`.

Ожидание: без откалиброванной подложки запуск recognition недоступен.

## 2. Локальное распознавание

1. Открыть `Распознать`.
2. Нажать `Распознать план`.
3. Наблюдать стадии обработки: подготовка → контуры → линии → стены → проёмы.
4. Дождаться review mode.

Ожидание:

- браузер не отправляет план во внешнюю AI-службу;
- UI не зависает намертво во время обработки;
- на Canvas появляются цветные recognition candidates;
- реальные стены проекта до применения не меняются;
- справа видны количество стен, проёмов, уверенных и требующих проверки элементов;
- на типовом сером плане застройщика движок использует строгий pass, а при слишком малом результате — scale-aware adaptive fallback вместо молчаливого `0`;
- если локальный CV всё же вернул пустой draft, UI явно сообщает об этом и предлагает `Повторить локально` или `Проверить с AI`.

## 3. Review mode

Проверить:

- выбор стены кликом;
- перетаскивание её endpoint-маркеров;
- `Принять` и `Отклонить`;
- `Принять уверенные`;
- выбор проёма;
- изменение `Неизвестный → Дверь/Окно`;
- rejected candidate становится визуально приглушённым;
- конфликтный candidate визуально отличается от high-confidence.

Ожидание: все эти изменения относятся только к recognition draft и не создают реальные стены/проёмы.

## 4. Persistence после F5

1. Принять несколько candidates и отклонить один.
2. Изменить endpoint стены.
3. Перезагрузить страницу.
4. Снова открыть `Распознать`.

Ожидание: тот же draft, решения пользователя и отредактированные координаты восстановлены.

При обновлении локального recognition engine старый draft другой версии должен помечаться `stale` и предлагать распознавание заново, а не молча восстанавливаться как актуальный результат.

## 5. Stale session

1. Имея сохранённый draft, заменить исходную подложку или выполнить новую метрическую калибровку.
2. Открыть recognition panel.

Ожидание:

- старый draft явно помечен как устаревший;
- применить его к новой revision нельзя;
- предлагается удалить draft или распознать заново.

Отдельно проверить изменение только opacity/visibility/позиции/поворота подложки: оно не должно делать session stale.

## 6. Safe apply и Undo

1. Принять несколько стен.
2. Оставить сомнительные элементы pending/rejected.
3. Нажать `Применить выбранное`.

Ожидание:

- создаются обычные структурированные стены Vlezet;
- pending/rejected элементы не применяются;
- неизвестный проём без классификации не применяется;
- вся операция появляется как одна запись истории.

Нажать один `Ctrl/Cmd + Z`.

Ожидание: весь применённый batch исчезает одним Undo.

Нажать Redo.

Ожидание: batch возвращается целиком.

## 7. Existing geometry safety

1. Вручную создать несколько стен поверх подложки.
2. Запустить recognition.
3. Найти совпадающие candidates.

Ожидание:

- существующие стены не дублируются;
- duplicate/conflict явно диагностируется;
- уже существующая совпадающая стена может использоваться как host для принятого распознанного проёма;
- никакой команды «заменить квартиру результатом AI» нет.

## 8. OpenRouter BYOK

### Быстрый путь

1. В review mode нажать `Проверить с AI`.
2. Ввести OpenRouter API key.
3. Сразу нажать `Анализировать`.

Ожидание: Vlezet сам получает совместимые vision + structured-output модели, выбирает первую из отсортированных по стоимости и запускает AI-анализ. Кнопка не должна выглядеть активной и при этом ничего не делать.

### Ручной выбор модели

1. Нажать `Выбрать модель вручную`.
2. Выбрать доступную vision-модель.
3. Запустить анализ.

Перед запуском UI должен явно сообщать, что изображение будет отправлено OpenRouter/выбранной модели.

Ожидание после успешного ответа:

- локальный draft не заменяется слепо;
- согласованные local+AI candidates объединяются;
- cloud-only candidates остаются reviewable;
- конфликтующие типы/геометрия требуют проверки;
- API key отсутствует в проекте и после закрытия диалога вводится заново.

## 9. Отмена и ошибки cloud

Проверить:

- закрытие/отмена во время AI-запроса;
- неверный API key;
- 402 / недостаточный баланс;
- rate limit;
- несовместимая модель или некорректный structured response.

Ожидание:

- существующий local draft не теряется;
- controller возвращается в review/error, но не остаётся навечно в `AI анализирует…`;
- проект не меняется;
- можно продолжить локальный review или ручную обводку.

## 10. Privacy / backup

1. После создания recognition draft экспортировать `.vlezet.json`.
2. По возможности открыть JSON текстовым редактором.

Ожидание:

- в backup есть проект и M4 reference raster;
- нет OpenRouter API key;
- нет `Authorization`/Bearer token;
- незавершённый recognition session не включён в backup.

Импортировать этот JSON как новый проект.

Ожидание: проект и подложка восстановлены, recognition draft отсутствует; распознавание можно запустить заново.

## 11. Duplicate/delete lifecycle

### Duplicate

Дублировать проект с существующим recognition draft.

Ожидание: копируется квартира и независимый reference asset, но незавершённый recognition session не копируется.

### Delete

Удалить проект с draft прямо с dashboard, в том числе после новой загрузки приложения.

Ожидание: проект, asset и recognition session удаляются без orphan data и без влияния на другие проекты.

## 12. Regression M0–M4

После recognition-проверки убедиться, что по-прежнему работают:

- pan/zoom и `Весь план`;
- стены и T-junctions;
- комнаты/площадь;
- двери/окна;
- мебель, resize/rotate/drag;
- fit statuses;
- autosave/reload;
- JSON import/export;
- PNG с подложкой и без;
- ручной `Начать обводку`.

## Acceptance result

M4.5 можно считать принятым вручную, когда полный путь проходит без потери данных:

```text
реальный план
→ калибровка
→ local recognition
→ review/edit
→ optional OpenRouter refinement
→ reconciliation
→ deterministic apply
→ Undo/Redo
→ reload
```

и при этом никакой внешний AI-ответ не становится геометрией проекта без явного пользовательского подтверждения и deterministic validation.
